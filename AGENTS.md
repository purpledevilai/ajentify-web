# Ajentify — coding agent reference

This file is the source of truth for building Ajentify stacks in this repository. It is generated by `ajentify init` and is intended to be read end-to-end before authoring or modifying `ajentify.json`.

> **You are an LLM coding agent reading this file.** The user will describe the agent stack they want in chat. Use this document to author `ajentify.json` and its supporting files (`prompts/`, `tools/`, `schemas/`) correctly, then run `ajentify deploy <stage> --plan` to validate before applying.

---

## 1. What is Ajentify?

Ajentify is a hosted platform for building conversational AI agents that can call tools and produce structured responses. You declare your agents, the tools they can call, and any Structured Response Endpoints (SREs) in a manifest. The Ajentify backend reconciles that manifest into a **stage** — a logical group of resources that the runtime addresses by `(stage, agent_logical_name)`.

The user interacts with deployed agents through:

- **POST /context** — start a conversation with an agent in a given stage.
- **POST /sre/{logical_name}** — invoke a structured response endpoint.
- **Web / mobile chat SDKs** — wrap those calls in a chat UI.

This repo is concerned with **authoring and deploying** the manifest, not with the runtime.

## 2. Stages

A stage is a named bucket of deployed resources within an organization. Stage names match `^[a-z][a-z0-9-]{0,62}$` — lowercase letters, digits, and dashes; must start with a letter; up to 63 characters. Examples: `staging`, `production`, `frontend-staging`, `backend-prod`.

Key properties:

- **The manifest is stage-agnostic.** A single `ajentify.json` describes the resources; the target stage is supplied separately as a CLI argument. The same manifest can be promoted between stages just by re-deploying with a different name.
- **Stages are created on first deploy.** You don't need to pre-create them.
- **Reconciliation is full.** Every deploy converges the stage to exactly the manifest. Resources not in the manifest are deleted.
- **Logical names are the reconciliation key.** The keys in `tools`, `sres`, and `agents` are *logical names* — stable identifiers within a stage. They must match `^[a-z][a-z0-9_]{0,62}$` (lowercase, digits, underscores; starts with a letter). Display names live in nested `name` fields and can be changed without breaking anything.
- **Resource IDs are server-managed.** They never appear in the manifest.
- **Idempotent.** Re-deploying the same manifest is a no-op.

## 3. The manifest shape

`ajentify.json` has three top-level maps:

```jsonc
{
  "$schema": "https://api.ajentify.com/docs/manifest-schema.json",
  "tools":  { "<logical_name>": { /* ManifestTool */ } },
  "sres":   { "<logical_name>": { /* ManifestSRE */ } },
  "agents": { "<logical_name>": { /* ManifestAgent */ } }
}
```

All three maps default to empty; you only include what you need.

The on-disk file is a **superset** of the wire schema served at <https://api.ajentify.com/docs/manifest-schema.json>. The `ajentify deploy` CLI inlines `*_file` references and strips them before POSTing; the body that hits `/deploy` matches the wire schema exactly.

### 3.1 Reconciliation order

The server applies operations in dependency order:

1. Create / update synthesized parameter definitions (from inline `input_schema` / `output_schema` on tools / SREs).
2. Create / update tools.
3. Create / update SREs.
4. Create / update agents.
5. Delete in reverse order (agents → SREs → tools → parameter definitions).

You don't need to think about parameter definitions — they're synthesized from the inline schemas you author on tools and SREs.

## 4. Resource reference

### 4.1 `ManifestTool`

A Tool is a server-side Python function (or a client-side stub) that an agent can call.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `name` | string | yes | Display name as shown to the agent. Normalized to snake_case server-side. |
| `description` | string | no | What the tool does. Shown to the agent — write it for an LLM consumer. |
| `input_schema` | object | no | JSON Schema (Draft 2020-12) for arguments. Omit if the tool takes no arguments. |
| `code` | string | no | Python source for a server-side tool. Omit for client-side tools. |
| `pass_context` | boolean | no, default `false` | When true, the tool receives the full context dict as an extra parameter. |
| `is_async` | boolean | no, default `false` | When true, the tool runs asynchronously and results are delivered later. |
| `is_client_side_tool` | boolean | no, default `false` | When true, the tool executes on the client (the chat SDK runs it) rather than on the server. |

Plus the file-reference helpers (resolved by the CLI):

- `code_file: "tools/sort_item.py"` — inlined into `code` (trailing newline trimmed).
- `input_schema_file: "schemas/sortItem.ts#SortItemInput"` — Zod export converted to JSON Schema, written to `input_schema`.

**Tool code shape.** Server-side tools are Python functions. The function name is irrelevant — only the input schema matters for what the agent sees. When `pass_context: true`, the signature includes a `context` parameter (positional or keyword) which gives the function access to the conversation context dict, including `context["user_defined"]` for authenticated user data passed in by the chat client.

```python
# tools/sort_item.py — pass_context: true
def sort_item(classification, context):
    user_defined = context.get("user_defined", {})
    # ... return any JSON-serializable value ...
    return {"bin": "yellow"}
```

The Python tool runtime auto-imports common modules (`json`, `requests`, etc.) — don't add `import` lines unless you need a less common module.

### 4.2 `ManifestSRE` (Structured Response Endpoint)

An SRE is a one-shot LLM call that produces output matching a JSON Schema. Useful for classification, extraction, summarization, etc. — anywhere you'd want "give me a typed value back" rather than a full conversation.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `name` | string | yes | Display name. |
| `description` | string | no | What the SRE does. |
| `output_schema` | object | **yes** | JSON Schema describing the structured output. Every SRE has structured output. |
| `is_public` | boolean | no, default `false` | When true, the SRE can be invoked without authentication. |
| `prompt_template` | string | yes | The prompt sent to the LLM. May contain placeholders. |
| `variable_names` | string[] | no | Names of placeholders in `prompt_template` to substitute at call time. |
| `model_id` | string | no | LLM model. Defaults to the org's default model. |

Plus file-reference helpers:

- `output_schema_file: "schemas/classify.ts#ClassifyOutput"` — Zod export converted to JSON Schema, written to `output_schema`.
- `prompt_template_file: "prompts/classify.md"` — inlined into `prompt_template`.

### 4.3 `ManifestAgent`

An Agent is a conversational LLM with access to a set of tools.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `name` | string | yes | Display name. |
| `description` | string | yes | Brief description of the agent's purpose. |
| `prompt` | string | yes | System prompt that defines the agent's behavior. |
| `is_public` | boolean | no, default `false` | When true, contexts can be created without authentication. |
| `agent_speaks_first` | boolean | no, default `false` | When true, the agent sends an initial message when a context is created. |
| `tools` | string[] | no | Tool references. Each entry is resolved in order: (1) a logical name from this manifest's `tools`, (2) a built-in tool ID from the registry (§7). A manifest logical name shadows a same-named registry ID. |
| `uses_prompt_args` | boolean | no, default `false` | When true, the prompt contains template arguments substituted per context. |
| `prompt_arg_names` | string[] | no | Names of the arguments referenced in the prompt. |
| `voice_id` | string | no | Voice ID for text-to-speech output. |
| `initialize_tool_id` | string | no | Tool to execute when a context is initialized. Same resolution rules as `tools`. |
| `model_id` | string | no | LLM model identifier. Defaults to the org's default. |

Plus file-reference helper:

- `prompt_file: "prompts/pod.md"` — inlined into `prompt`.

**`uses_prompt_args` / `prompt_arg_names`.** When the agent needs per-conversation data baked into its prompt (e.g. the current user's profile), set `uses_prompt_args: true` and list the placeholder names in `prompt_arg_names`. The chat client provides values for those names when starting a context, and the runtime substitutes them into the prompt before the first turn.

## 5. File-reference helpers (the CLI's value-add)

The wire schema requires you to inline every prompt, every Python tool body, and every JSON Schema into one JSON file. That's painful to author and review. `ajentify deploy` lets you keep each in its own file:

| Resource | Inline field | File reference field | Notes |
| --- | --- | --- | --- |
| Agent | `prompt` | `prompt_file: "prompts/pod.md"` | Plain text / markdown. |
| Tool | `code` | `code_file: "tools/sort_item.py"` | Python source. Trailing newline trimmed. |
| Tool | `input_schema` | `input_schema_file: "schemas/sortItem.ts#SortItemInput"` | TS Zod export → JSON Schema. |
| SRE | `output_schema` | `output_schema_file: "schemas/classify.ts#ClassifyOutput"` | TS Zod export → JSON Schema. |
| SRE | `prompt_template` | `prompt_template_file: "prompts/classify.md"` | Plain text. |

Rules:

- The `#ExportName` fragment on `*_schema_file` picks a named export. Omit it to use the default export.
- Setting both the inline field and the `*_file` form for the same resource is a hard error — pick one.
- Paths are resolved relative to the `ajentify.json` file's directory.
- The CLI strips the `*_file` fields after inlining; only canonical fields are sent to the server.

### 5.1 Authoring Zod schemas

`*_schema_file` files are TypeScript files that export a Zod schema. The CLI loads them at deploy time via `jiti` and converts them with `z.toJSONSchema` (Zod v4) or `zod-to-json-schema` (Zod v3).

```ts
// schemas/sortItem.ts
import { z } from "zod";

export const SortItemInput = z.object({
  classification: z
    .enum(["cardboard", "container", "food", "garbage", "garden", "glass", "soft_plastic"])
    .describe("Material category of the item."),
});
```

Use `.describe(...)` liberally — the descriptions show up in the JSON Schema and the agent uses them to decide how to call the tool. The schema's top-level `additionalProperties: false` and `required` arrays are produced automatically.

## 6. Recommended project layout

```
my-project/
├── ajentify.json
├── AGENTS.md                ← this file
├── .env                     ← AJENTIFY_API_KEY=...
├── prompts/
│   ├── pod.md
│   └── classify.md
├── tools/
│   ├── sort_item.py
│   └── show_bin.py
└── schemas/
    ├── sortItem.ts
    └── classify.ts
```

Nothing about the CLI requires this layout — paths in `*_file` fields can point anywhere — but this is what users / agents will recognize on sight.

## 7. Built-in tool registry

An agent can reference these tool IDs in its `tools` list without declaring them in this manifest. Use them whenever you can — they're maintained centrally and come with auth integrations already wired up.

- **Utility:** `get_time`, `think`
- **Events:** `pass_event`
- **Memory:** `append_memory`, `read_memory`, `write_memory`, `delete_memory`, `view_memory_shape`, `open_memory_window`
- **Web Search:** `web_search`, `view_url`
- **Data Window:** `open_data_window`
- **Gmail:** `list_emails`, `get_email`, `send_email`, `set_email_read_status`, `create_draft`, `list_drafts`, `get_draft`, `update_draft`, `send_draft`, `delete_draft`, `list_labels`, `create_label`, `delete_label`, `modify_email_labels`, `archive_email`, `trash_email`, `untrash_email`, `delete_email`
- **Outlook:** `list_outlook_emails`, `get_outlook_email`, `send_outlook_email`, `set_outlook_email_read_status`, `create_outlook_draft`, `list_outlook_drafts`, `get_outlook_draft`, `update_outlook_draft`, `send_outlook_draft`, `delete_outlook_draft`, `list_outlook_folders`, `create_outlook_folder`, `delete_outlook_folder`, `move_outlook_email`, `modify_outlook_email_categories`, `reply_outlook_email`, `reply_all_outlook_email`, `create_outlook_reply_draft`, `create_outlook_reply_all_draft`, `archive_outlook_email`, `trash_outlook_email`, `untrash_outlook_email`, `delete_outlook_email`

A manifest-declared tool whose logical name matches a registry ID *shadows* the registry tool. Pick distinct names unless that's intentional.

## 8. End-to-end example

A stack with one agent that uses a custom server-side tool plus a built-in tool, and one SRE.

### `ajentify.json`

```jsonc
{
  "$schema": "https://api.ajentify.com/docs/manifest-schema.json",
  "tools": {
    "sort_item": {
      "name": "sort_item",
      "description": "Decide which kerbside bin a classified item belongs in.",
      "input_schema_file": "schemas/sortItem.ts#SortItemInput",
      "code_file": "tools/sort_item.py",
      "pass_context": true
    }
  },
  "sres": {
    "classify": {
      "name": "Classify item",
      "description": "Classify a free-form item description into one of seven material categories.",
      "output_schema_file": "schemas/classify.ts#ClassifyOutput",
      "prompt_template_file": "prompts/classify.md",
      "variable_names": ["ITEM_DESCRIPTION"]
    }
  },
  "agents": {
    "pod": {
      "name": "Pod",
      "description": "Helps users sort waste into the correct bin.",
      "prompt_file": "prompts/pod.md",
      "agent_speaks_first": true,
      "tools": ["sort_item", "get_time"]
    }
  }
}
```

### `schemas/sortItem.ts`

```ts
import { z } from "zod";

export const SortItemInput = z.object({
  classification: z
    .enum(["cardboard", "container", "food", "garbage", "garden", "glass", "soft_plastic"])
    .describe("The material category of the item to sort."),
});
```

### `schemas/classify.ts`

```ts
import { z } from "zod";

export const ClassifyOutput = z.object({
  category: z
    .enum(["cardboard", "container", "food", "garbage", "garden", "glass", "soft_plastic"])
    .describe("The material category of ITEM_DESCRIPTION."),
  confidence: z.number().min(0).max(1).describe("How confident, 0..1."),
});
```

### `prompts/pod.md`

```
You are Pod, an assistant that helps users sort waste into the correct kerbside bin.

When the user describes an item:
1. Call `sort_item` with the right material category.
2. Tell them which bin to use.

Use `get_time` if the user asks about collection schedules.
```

### `prompts/classify.md`

```
Classify the following item into one of the seven categories.

ITEM: ITEM_DESCRIPTION
```

### `tools/sort_item.py`

```python
def sort_item(classification, context):
    user_defined = context.get("user_defined", {})
    user_data = json.loads(user_defined.get("user_data", "{}"))
    bins = user_data.get("bin_system", {}).get("bins", [])

    flag_for = {
        "cardboard": "acceptsCardboard",
        "container": "acceptsContainers",
        "food": "acceptsFood",
        "garbage": "acceptsGarbage",
        "garden": "acceptsGarden",
        "glass": "acceptsGlass",
        "soft_plastic": "acceptsSoftPlastics",
    }
    flag = flag_for.get(classification)
    if not flag:
        return {"error": f"Unknown classification {classification!r}"}

    for b in bins:
        if b.get(flag):
            return {"bin_type": b.get("type"), "bin_color": b.get("appearance")}
    return {"no_bin_found": True, "classification": classification}
```

## 9. CLI usage

```bash
ajentify init                       # scaffold ajentify.json, AGENTS.md, .env.example
ajentify deploy staging --plan      # dry-run: show diff without applying
ajentify deploy staging             # apply
ajentify deploy production          # same manifest, different stage (promotion)
```

Useful flags:

- `--manifest <path>` — defaults to `./ajentify.json`.
- `--org-id <id>` — pin a specific org. Defaults to the API key's first org.

The CLI reads `AJENTIFY_API_KEY` from `process.env`, or from the nearest `.env` walking up from the manifest's directory.

## 10. Workflow for coding agents

When the user asks you to build or modify a stack:

1. **Read the user's intent carefully.** What does the agent do? What tools does it need? Does it produce structured output anywhere (SRE) or only converse (Agent)?
2. **Reuse built-in tools** from §7 wherever possible. Don't reimplement `web_search`, `get_time`, Gmail / Outlook / Calendar tools, etc.
3. **Author files in this order:**
   - Tool Python files in `tools/`.
   - Zod schemas in `schemas/` (with rich `.describe(...)` calls).
   - Prompt files in `prompts/`.
   - The `ajentify.json` manifest tying them together.
4. **Pick logical names carefully.** They're stable reconciliation keys — renaming a logical name = delete + recreate.
5. **Validate with `--plan`** before applying. The plan output shows exactly which resources will be created / updated / deleted.
6. **Apply** with `ajentify deploy <stage>`.

## 11. Troubleshooting

- **"prompt_file ... points to ... but ... does not exist"** — the path is wrong or relative-to-wrong-base. Paths resolve relative to the `ajentify.json` file.
- **"... has both 'prompt' and 'prompt_file'; pick one"** — remove whichever you don't want.
- **"references unknown tool 'foo'"** — the agent's `tools` entry doesn't match any manifest logical name or registry ID. Fix the name or declare the tool.
- **"Invalid logical_name 'Foo'"** — keys must match `^[a-z][a-z0-9_]{0,62}$` (lowercase, digits, underscores, starts with a letter).
- **"Invalid stage 'Foo'"** — stage names follow the same rule but use dashes instead of underscores: `^[a-z][a-z0-9-]{0,62}$`.
- **HTTP 401** — `AJENTIFY_API_KEY` is missing or wrong.
- **HTTP 403** — the key isn't a member of the `--org-id` you passed.
